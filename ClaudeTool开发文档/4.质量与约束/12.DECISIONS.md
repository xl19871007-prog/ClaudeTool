# DECISIONS — 架构决策日志（ADR）

> 重要技术决策必须记一条 ADR。变更前先开新 ADR，确认后再改代码与文档。

---

## 什么时候记一条决策

记 ADR：
- 在 A / B / C 多方案间做了选择
- 选了"不直觉"的方案，将来会被人质疑
- 决定**不**做某件事
- 引入影响多个模块的约定

不记 ADR：
- 琐碎 code 级选择
- 临时修复
- 已在其他文档明确的东西

### ADR-001: 选用 Tauri 2 而非 Electron 作为桌面壳
- **日期**：2026-04-20
- **状态**：已接受
- **背景**：MVP 的核心定位是「轻量、对小白友好」，需要在内存占用、包体大小、启动速度上明显优于现有 Electron GUI（如官方 Claude Code Desktop）。
- **选项**：
  - A：Electron + node-pty（生态成熟、文档丰富、案例多）
  - B：Tauri 2 + tauri-plugin-pty（Rust + WebView，包小、内存省）
  - C：原生 Win32（C++/Rust + Win32 GUI 库）— 跨平台成本高
- **决定**：选 B（Tauri 2）
- **理由**：
  - 2026 年公开数据：Tauri 空闲内存 30–60MB vs Electron 200–300MB；包体小约 96%；启动快约 40%。
  - 与"轻量小白工具"产品定位高度匹配。
  - Opcode（21K Star）已用 Tauri 2 验证可行。
  - Rust 生态有 `tauri-plugin-pty`、`notify`、`reqwest` 等可直接用。
- **后果**：
  - 正面：用户体验更好；潜在差异化亮点。
  - 负面：团队需补 Rust 知识；某些 npm 生态库无法直接用；Linux 端 WebKitGTK 在部分发行版需用户装依赖。
  - 额外事项：CI 需配 Rust toolchain；文档要写明 Linux 用户需装 `webkit2gtk`。
- **相关**：MARKET_RESEARCH.md §3 需求 2 / TECH_STACK.md

### ADR-002: 暂缓 Token 配额监控功能（PRD F3）
- **日期**：2026-04-20
- **状态**：已接受
- **背景**：用户原始需求 #3 希望实时展示「Token 总量、剩余量、下次刷新时间」。调研发现 Anthropic 未提供官方配额查询 API。
- **选项**：
  - A：用第三方做法估算（解析本地 jsonl + 让用户手动选套餐 + P90 历史推算）
  - B：暂缓，等官方 API
  - C：只显示已消耗，不显示剩余与刷新（去掉一半价值）
- **决定**：选 B
- **理由**：
  - 估算 vs 实际可能误差很大，对小白用户 = 错误信息 = 损坏信任。
  - 与 CLAUDE.md 核心原则 #1「数据准确性优先」直接冲突。
  - 现有第三方工具（ccusage、Usage Monitor）即便很努力也只能给"参考"，与本工具"权威助手"的定位不一致。
- **后果**：
  - 正面：守住准确性底线；省下 MVP 的 1–2 周开发预算可投到其他需求。
  - 负面：对重度用户的吸引力降低（他们可继续用 ccusage CLI）。
  - 额外事项：监控 Anthropic API 路线图，官方放出配额接口立刻启动 F3。
- **相关**：MARKET_RESEARCH.md §3 需求 3 / PRD F3

### ADR-004: 不主动推广任何形式的区域限制规避
- **日期**：2026-04-20
- **状态**：已接受（"提供代理设置项"实施细节已被 ADR-013 修订删除；主决策"不主动推广绕过"不变）
- **背景**：Anthropic 自 2025-09 起强化区域限制，中国大陆 IP 直接被拒。中文用户最痛点之一是"网络不通"。如何处理是产品方向问题。
- **选项**：
  - A：内置代理工具 / 推荐 VPN 列表
  - B：完全不提，网络不通时只显示原始报错
  - C：温和提示「请确认网络可访问 Anthropic」+ 提供让用户填代理的设置项（不评价代理来源）
- **决定**：选 C
- **理由**：
  - A 直接违反 Anthropic 服务条款，可能导致项目被点名甚至下架。
  - B 对中文用户太冷漠，等于放弃帮助。
  - C 是"工具中立"——本工具不参与决策，由用户自己决定怎么解决网络问题，我们只负责把用户配好的代理透传给 PTY。
- **后果**：
  - 正面：合规风险最小；同时实质解决用户问题（让代理生效到子进程）。
  - 负面：完全不熟悉代理的小白需自己学习"代理是什么"，门槛仍在。
  - 额外事项：错误文案、README、官网都要严格审查，避免出现敏感措辞；代理设置 UI 不可暗示推荐任何具体工具。
- **相关**：CLAUDE.md #2 / CONSTRAINTS.md §9 / PRD F4

### ADR-005: 嵌入式终端走 Rust 端 `tauri-plugin-pty`
- **日期**：2026-04-20
- **状态**：已接受
- **背景**：实现嵌入式终端需 PTY 支持。可选方案多。
- **选项**：
  - A：Node sidecar 跑 `node-pty`（成熟，Electron 圈最常用）
  - B：Rust 端 `tauri-plugin-pty`（Win 走 ConPTY，*nix 走 forkpty）
  - C：自写 Rust PTY 包装（过度自制）
- **决定**：选 B
- **理由**：
  - 与 Tauri 主语言一致，避免双运行时。
  - 不需打包 Node sidecar，减少 50MB+ 包体。
  - PTY I/O 在 Rust 端零 GC，吞吐与延迟更好。
- **后果**：
  - 正面：包体与性能符合 CONSTRAINTS。
  - 负面：`tauri-plugin-pty` 文档与示例不如 `node-pty` 丰富，需投入学习。
  - 额外事项：Win 上 ConPTY 的颜色/UTF-8 问题历史上常见，需做集成测试。
- **相关**：TECH_STACK.md §1 / ARCHITECTURE.md `pty_manager`

### ADR-006: Skills 推荐 MVP 用静态种子，V2 接动态市场
- **日期**：2026-04-20
- **状态**：已接受（"种子来源人工 curate"实施细节已被 ADR-015 修订为"直接用 anthropics/skills 官方仓库"；主决策"MVP 用静态种子"不变）
- **背景**：PRD F5 要展示「下载量 Top 10 未安装的 Skills」。无官方下载量 API，第三方 marketplace（claudemarketplaces.com、PolySkill 等）无稳定开放接口。
- **选项**：
  - A：MVP 直接爬 marketplace 网站
  - B：MVP 内嵌人工筛选的种子列表 + 人工标注的"近似 Star 数"，V2 再接动态
  - C：MVP 直接调 GitHub Search API（受限于配额）
- **决定**：选 B
- **理由**：
  - 爬虫脆弱、易被改版打断；marketplace 网站可能不允许自动化访问。
  - 人工筛选保证质量，给小白用户「精挑细选」的感觉，而非「全网随便列」。
  - 数据条目数 30 即可（推荐 Top 10 是其子集），维护成本可接受。
- **后果**：
  - 正面：MVP 时间内能交付；推荐质量可控。
  - 负面：Star 数会过时；新 Skill 不会自动出现。
  - 额外事项：种子文件必须含 `recordedAt` 字段，UI 展示「数据更新于：XX」；版本发布 checklist 加入「同步种子」一项。
- **相关**：DATA_MODEL.md `SkillSeed` / PRD F5

### ADR-007: 对话方向引导（PRD F8）走规则引擎而非 Hook 方案
- **日期**：2026-04-20
- **状态**：已接受
- **背景**：用户希望帮小白避免把 AI"带歪"。三种实现路线：规则引擎、Coach 大模型、Claude Code Hook。
- **选项**：
  - A：规则引擎（监听 jsonl，命中规则给侧栏提示卡片）
  - B：调 Haiku 当 Coach（智能但要 API key、要计费）
  - C：注册 UserPromptSubmit Hook 改用户输入
- **决定**：MVP 不做（推迟 V2）；V2 走方案 A 起步，后续可选方案 B
- **理由**：
  - 方案 C 直接修改用户输入会非常突兀，体验差。
  - 方案 B 增加用户门槛（需另配 API key），与小白定位冲突。
  - 方案 A 零成本、零门槛、本地运行、规则可控；MVP 内做不完，留给 V2 起步。
- **后果**：
  - 正面：MVP 简化，体验路径不被打断。
  - 负面：早期不能直接展示这个差异化亮点。
  - 额外事项：MVP 实现 `history_parser::watch_active_session`（V2 用，但接口先在 MVP 留出避免重构）。
- **相关**：MARKET_RESEARCH.md §3 需求 8 / PRD F8 / ARCHITECTURE.md `history_parser`

### ADR-009: MVP 包含一键安装 Claude Code，仅走 Win 原生安装包路径
- **日期**：2026-04-20
- **状态**：已接受（覆盖 ADR-008 之前 PRD F1 隐含的"MVP 不做一键安装"范围）
- **背景**：PRD 第 1 轮评审 Q1.2 用户选 B：MVP 直接做应用内一键安装。原方案"跳转下载页让用户自己装"对小白可能仍卡在 PATH/UAC/不知道怎么回来。
- **选项**：
  - A：维持原方案（跳转下载链接），最小工期
  - B：MVP 做一键安装，仅走 Anthropic 官方 **Windows 原生安装包**路径（下载 → 校验签名 → spawn 安装包 → 让 Win 弹 UAC → 验证）
  - C：MVP 做更彻底的一键安装，包含 npm 路径与 Node 自动安装（处理 Node 缺失场景）
- **决定**：选 B
- **理由**：
  - Anthropic 自 2026 起官方提供 Windows 原生安装包，PATH/UAC 由它自己处理，本工具只需当下载与启动器
  - C 涉及 Node 检测/下载/安装/PATH 写入，分支多、失败点多，不符合 MVP 范围
  - MVP 已锁定仅 Win（ADR-003），简化了平台分支
- **后果**：
  - 正面：小白完成率显著提升（B 的核心价值）；体验闭环更完整
  - 负面：MVP 工期 +1.5–2 周（8 → 10 周）；首次发布会触发 SmartScreen 警告（无 EV 证书）；用户拒绝 UAC 走回退路径
  - 额外事项：
    - 必须校验下载的安装包数字签名（CONSTRAINTS §3）
    - ROADMAP 增加 M4「一键安装」里程碑
    - PRD F1 重写、ARCHITECTURE `env_checker` 模块新增 `install_claude_native` 接口
    - npm 路径与 Node 自动安装推迟到 V1.1
- **相关**：PRD F1 / ARCHITECTURE `env_checker` / ROADMAP M4 / CONSTRAINTS §3

### ADR-011: jsonl 与 Claude Code 版本兼容策略
- **日期**：2026-04-20
- **状态**：已接受
- **背景**：第 2 轮评审 Q2.1。本工具依赖 `~/.claude/` 下文件结构与 jsonl schema，Anthropic 升级 CLI 时可能改字段名 / 新增字段，导致历史会话面板、Skills 扫描出错。
- **选项**：
  - A：内置已测版本范围常量（`SUPPORTED_CCODE_RANGE`），超出弹兼容性横幅
  - B：完全宽松解析，不告警
  - C：A + V2 加「用户主动点按钮」上报兼容性问题到 GitHub Issue
- **决定**：选 C（MVP 实现 A 部分，V2 加上报功能）
- **理由**：
  - A 让用户对兼容性风险有预期，比静默失败强。
  - 上报功能让我们快速发现新版本破坏；用户主动触发（不是后台采集），不违反 CLAUDE.md #3「不收集后台遥测」。
- **后果**：
  - MVP 必须维护版本白名单常量；每次发版前回归 + 更新（已加入 ROADMAP §发布前 Checklist）
  - UI_UX 加"兼容性横幅"模式
  - V2 在横幅上加「提交反馈」按钮 → 打开预填好的 GitHub Issue 页面
- **相关**：CONSTRAINTS §6 / UI_UX §6 / ARCHITECTURE history_parser

### ADR-012: WebView2 用 Bundled 模式打包
- **日期**：2026-04-20
- **状态**：已接受
- **背景**：第 2 轮评审 Q2.3。Tauri 在 Windows 用 WebView2 渲染前端，分发策略需选择。
- **选项**：
  - A：Bundled —— 安装包内嵌完整 WebView2 离线运行时（+150–180MB）
  - B：Downloader —— 安装包内嵌轻量引导器，首次安装时联网下载 WebView2
  - C：System —— 不打包 WebView2，要求用户自己装
- **决定**：选 A
- **理由**：
  - 离线可装对小白友好（避免"装一半网断了怎么办"）
  - 老 Win10（无 WebView2）兼容性最优
  - 一次性下载完所有依赖，不会半途失败
- **后果**：
  - 正面：安装成功率最高；可在网络受限环境装
  - 负面：安装包从 30MB 暴增到 ~180–230MB；用户首次下载耗时长
  - 额外事项：PRD §6 成功指标和 CONSTRAINTS §2 资源占用已同步更新
- **相关**：PRD §6 / CONSTRAINTS §2 / TECH_STACK §2

### ADR-013: MVP 不内置代理配置（局部修订 ADR-004）
- **日期**：2026-04-20
- **状态**：已接受（局部修订 ADR-004）
- **背景**：第 2 轮评审 Q2.4。原 ADR-004 决定"不主动推广绕过 + 提供代理设置项让用户自填"。Q2.4 进一步收紧：网络问题完全由用户在 OS 层面（VPN）解决，本工具不提供任何代理相关 UI 与配置。
- **选项**：
  - A：保留代理输入框（原 ADR-004 实施细节）
  - B：完全不提供代理相关 UI，用户用 OS 层 VPN
  - C：A + 加教程（边界风险大）
- **决定**：选 B
- **理由**：
  - OS 层 VPN 影响所有进程（含 PTY 子进程），不需要本工具显式注入 `HTTPS_PROXY`，技术上可行。
  - 立场更中性：本工具完全不参与网络配置，进一步降低合规风险。
  - 简化 AppConfig schema 与 UI（少一个设置面板项与一组字段）。
- **后果**：
  - 删除 `AppConfig.proxy.*` 字段
  - PRD F4 改为「不内置代理配置」
  - ARCHITECTURE `pty_manager` 删除"从 config 拿代理"的注入逻辑（保留 `env_overrides` 通用扩展点）
  - UI_UX 删除 SettingsDialog 中的代理输入项
  - 失去 MARKET_RESEARCH §3 提到的"自动注入代理到 PTY"差异化亮点（接受这一损失）
  - ADR-004 主决策不变；只删除其中"提供代理设置项"的实施细节

### ADR-014: 历史会话解析采用流式 + 后台异步精确化
- **日期**：2026-04-20
- **状态**：已接受
- **背景**：第 3 轮评审 Q3.3。CONSTRAINTS 要求"100 个会话扫描 ≤ 300ms"，但单个 jsonl 可能数 MB（长会话），全读不可行；只读头部又可能漏掉 Claude Code 在末尾才写的字段。
- **选项**：
  - A：只读文件头 N 行（最快，可能漏字段）
  - B：读全文件（准确但慢，100 × 数 MB I/O）
  - C：流式读，遇到关键字段就停（折中）
  - D：C + 后台异步精确化 `turnCount` 等慢字段（首屏快显示，详情慢补）
- **决定**：选 D
- **理由**：
  - 用户体验最优：选完文件夹立刻看到列表，详情逐条刷新
  - 满足 ≤ 300ms 首屏指标
  - 实现复杂度可控（用 Tauri event 推送 patch，前端 store 局部更新）
- **后果**：
  - 正面：首屏快、最终数据准
  - 负面：实现复杂度比 A 高；前端要处理"数字从 ~ 变 12 轮"的过渡（用 skeleton 或浅灰过渡色）
  - 额外事项：
    - `history_parser` 暴露 `list_sessions_quick` + `refine_session_metadata` 双接口
    - `SessionMeta.turnCount` 文档标注"流式估值，后续精确化"
    - 前端组件需实现局部更新订阅
- **相关**：ARCHITECTURE history_parser / DATA_MODEL SessionMeta / CONSTRAINTS §1

### ADR-016: Skills 推荐数据按 plugin 维度建模（修正 ADR-015 实施细节）
- **日期**：2026-04-20
- **状态**：已接受（修正 ADR-015 在数据模型层面的隐含假设）
- **背景**：M6 实测发现，初版 `seed-skills.json` 把 `pdf`、`docx`、`code-review` 等当作独立 plugin，结果 `claude plugin install pdf@anthropics/skills` 全部失败：
  1. 实际上 Anthropic 官方 marketplace 只有 3 个 plugin（`document-skills` / `example-skills` / `claude-api`），单个 skill 不能直接安装；
  2. marketplace 的注册名是 `anthropic-agent-skills`（来自 marketplace.json `name` 字段），而非 GitHub repo 路径 `anthropics/skills`；
  3. 我之前的种子还**捏造了不存在的 skill**（如 git-worktree、test-writer），属于幻觉数据。
- **选项**：
  - A：保持 skill 维度，给每条种子加上"所属 plugin"字段（数据冗余，仍违反"安装单位是 plugin"的事实）
  - B：改为 plugin 维度，每条种子是一个 plugin，列出 bundledSkills；安装命令模板用真实 marketplaceId
- **决定**：选 B
- **理由**：
  - 与 Claude Code 实际命令模型一致（`plugin install` 接受 plugin 名）
  - 杜绝"展示了一个 skill 但不能装"的伪命题
  - 推荐列表条目数变少（3 而非 15）但每条信息密度更高（含 N 个 skill 的预览）
- **后果**：
  - `seed-skills.json` 由 15 条 skill 重写为 3 条 plugin（覆盖官方 marketplace 全集）
  - `SkillMeta` 增加 `bundledSkills?: string[]` 与 `marketplaceId?: string` 字段
  - SkillsPanel 推荐 Tab 渲染 plugin 卡片，详情页显示 bundled skills 标签 + Win git-bash 提示
  - 数据更准但失去"列 N+ 条候选"的视觉规模感（接受）
  - 教训沉淀到 DOMAIN.md「业务规则」+ 术语表
- **相关**：ADR-015 / DOMAIN.md / `seed-skills.json` / `skills_scanner.rs` / `SkillsPanel.tsx`

### ADR-015: Skills 种子 MVP 直接用 anthropics/skills 官方仓库（修订 ADR-006）
- **日期**：2026-04-20
- **状态**：已接受（修订 ADR-006）
- **背景**：第 4 轮评审 Q4.3。原 ADR-006 决定 MVP 用"人工筛选种子 + 人工标注 stars"。Q4.3 选 C 进一步简化：直接列 `anthropics/skills` 官方 ~30 条作为种子，省下人工 curate 与审核的工作量。
- **选项**：
  - A：人工 curate 30 条社区高赞（原 ADR-006 决定）
  - B：Claude 起初稿 + 人工审（折中）
  - C：直接用 anthropics/skills 官方仓库（最省事，但缺乏多元）
- **决定**：选 C
- **理由**：
  - 官方仓库本身就是"权威推荐"，对小白用户友好
  - 省去 curate 与维护争议（"凭什么选这个不选那个"）
  - 与 ADR-006 主旨一致（用静态种子而非动态市场），仅来源不同
- **后果**：
  - 正面：交付物简化（直接 `git submodule add anthropics/skills` 或人工同步 README 列表）；权威性高
  - 负面：失去"社区热门"维度；可推荐项数有限（约 30 条，与 PRD F5"Top 10"自然契合）；如官方仓库改名/变结构需跟版
  - 额外事项：
    - `SkillSeed.stars` 字段意义弱化（官方仓库无 star 概念），改为按官方文档顺序排序
    - `recordedAt` 字段保留，记录最近一次同步官方仓库的时间
    - ROADMAP 发布前 Checklist 中"同步 30 条 Skills 种子"改为"同步 anthropics/skills 仓库到 seed-skills.json"
- **相关**：ADR-006（修订）/ PRD F5 / DATA_MODEL `SkillSeed`
